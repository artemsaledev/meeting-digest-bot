# Пользовательская инструкция: база знаний по крупным задачам

## Назначение

База знаний собирает и удерживает историю крупных задач: обсуждения, demo,
Loom-разборы, публикации в Telegram, связанные Bitrix-задачи, комментарии,
чеклисты, решения, требования и открытые вопросы.

В базу должны попадать не daily-планы, а рабочие артефакты с тегами:

- `#task_discussion` - обсуждение задачи, требований, вариантов реализации.
- `#task_demo` - демонстрация результата, фидбек, уточнения после показа.

Итоговая цель: из накопленных объектов можно спрашивать информацию по системам
и функциональности, а затем формировать инструкции, ТЗ, спецификации,
acceptance criteria и support-ответы.

## Где находится источник правды

Основной источник правды - GitHub repository `artemsaledev/company-knowledge`.

Notion используется как удобная поверхность для просмотра и ручных правок.
NotebookLM, RAG-инструменты и agent bundles получают экспорт из canonical базы.

Правило простое: Git/JSON - источник правды, Notion - редактор и витрина.

## Как задача попадает в базу знаний

1. В рабочем процессе появляется крупное обсуждение или demo.
2. В источнике должен быть тег `#task_discussion` или `#task_demo`.
3. Loom/transcript обрабатывается существующим pipeline.
4. Digest публикуется в Telegram.
5. Публикация регистрируется в системе.
6. Knowledge pipeline создает или обновляет `task_case`.
7. Из `task_case` автоматически пересобираются:
   - `Systems`
   - `Features`
   - `Instructions`
8. Обновляются индексы, RAG, Notion и external exports.

## Основные типы объектов

`Task Case` - конкретная крупная задача или накопленная история по задаче.
Содержит источники, требования, решения, acceptance criteria, demo feedback и
открытые вопросы.

`System` - сводка по системе, например Bitrix или AIcallorder.

`Feature` - сводка по функциональной области внутри системы.

`Instruction` - прикладная инструкция, собранная из подтвержденных task cases.

## Как спрашивать базу знаний в Telegram

Главный сценарий работы - через Telegram. Команды остаются как точный
управленческий контур, но целевой интерфейс должен быть проще: ты пишешь или
наговариваешь запрос, выбираешь действие кнопкой, а бот сам вызывает нужный
workflow базы знаний.

Базовые действия:

- задать вопрос по функциональности;
- сформировать пользовательскую инструкцию;
- сформировать ТЗ/спецификацию/acceptance criteria;
- посмотреть предложенные изменения;
- дать корректировку по инструкции или знанию;
- собрать bundle для NotebookLM или другого AI-инструмента;
- проверить состояние базы.

В группе, где подключен бот, уже доступны команды:

```text
kb health
kb ask <вопрос>
kb proposals
kb diff <id>
kb approve <id>
kb reject <id>
kb apply <id>
```

Примеры:

```text
kb health
kb ask как сейчас работает синхронизация чеклистов в Bitrix?
kb ask какие требования уже подтверждены по AIcallorder?
kb ask сформируй пользовательскую инструкцию по публикации Loom-дайджеста
```

`kb health` показывает короткое состояние базы: pending proposals, RAG chunks,
примерный usage и качество данных.

`kb ask` сначала использует RAG с внешними embeddings/LLM. Если внешний AI не
настроен, команда может fallback-нуться на локальный lexical search.

## Целевой сценарий с голосом и кнопками

Ты можешь написать или наговорить боту:

```text
Хочу понять, как сейчас работает публикация Loom-дайджеста в Telegram и Bitrix.
```

И выбрать кнопку:

- `Ask`
- `Instruction`
- `Spec`
- `Export`
- `Proposals`
- `Health`

Бот должен:

1. Распознать голос в текст, если вход был аудио.
2. Понять намерение по кнопке или по смыслу сообщения.
3. Найти релевантные knowledge objects через RAG.
4. Собрать ответ, инструкцию, ТЗ, proposal или export.
5. Вернуть результат прямо в Telegram.
6. Для файлов - приложить архив или написать точный путь/что загрузить во
   внешний AI-инструмент.

Notion остается как дополнительная поверхность для просмотра и ручного
редактирования, но не как основной способ работы.

## Как работать с Notion

В Notion можно читать и вручную править страницы базы знаний. Важно: ручная
правка Notion не меняет canonical JSON напрямую.

Правильный цикл такой:

1. Ты редактируешь страницу в Notion.
2. Import job находит отличие от canonical версии.
3. Система создает draft proposal.
4. Бот присылает уведомление в Telegram.
5. Ты смотришь diff.
6. Если правка верная - approve/apply.
7. Canonical JSON обновляется, затем пересобираются Notion/RAG/export.

Это защищает базу от случайного удаления блоков, неполного Notion read и
ошибочных ручных правок.

## Как принимать или отклонять ручные правки

Посмотреть список:

```text
kb proposals
```

Посмотреть diff:

```text
kb diff <object_id>
```

Одобрить:

```text
kb approve <object_id>
```

Применить:

```text
kb apply <object_id>
```

Отклонить:

```text
kb reject <object_id>
```

После применения proposal canonical объект обновляется, а pipeline пересобирает
производные объекты и индексы.

## Как формировать инструкции, ТЗ и спецификации

Запросы лучше задавать в разрезе системы, функциональности и нужного формата.

Хорошие формулировки:

```text
kb ask сформируй пользовательскую инструкцию по работе с чеклистами Bitrix
kb ask составь техническую спецификацию по публикации Loom-дайджестов в Telegram
kb ask какие acceptance criteria уже подтверждены для AIcallorder?
kb ask какие открытые вопросы остались по интеграции Notion и базы знаний?
```

Если нужен документ для внешнего AI, используется export bundle:

- NotebookLM source bundle - markdown sources для загрузки в NotebookLM.
- Machine bundle - JSON/context для API и агентов.
- Prompt workspace - готовые prompt-шаблоны для корректировок и генерации.

## Как корректировать knowledge object промптами

Рекомендуемый сценарий:

1. Сначала спросить базу:

```text
kb ask что известно по <система>/<функция>?
```

2. Сформулировать корректировку:

```text
В объекте <object_id> нужно уточнить, что demo-решение заменяет старое обсуждение.
Старое требование не удалять, а пометить как superseded.
```

3. Внести правку через Notion или proposal workflow.
4. Проверить diff.
5. Approve/apply.
6. Снова спросить `kb ask`, чтобы убедиться, что ответ изменился корректно.

Целевой Telegram-сценарий для корректировки:

```text
Эта инструкция неверная: после demo решили не перезаписывать старые пункты,
а добавлять новый Point N и отдельный checklist. Обнови знание и покажи proposal.
```

Бот должен сам найти релевантный объект, создать revision proposal, показать
короткое резюме изменений и предложить approve/reject/apply.

## Что не нужно класть в базу знаний

Не нужно добавлять:

- daily-планы без крупного решения;
- короткие статусы без требований или решений;
- временные напоминания;
- личные заметки без связи с системой или функциональностью;
- сырые расшифровки без summary, если они не привязаны к task discussion/demo.

## Статусы объектов

`draft` - объект создан или изменен, но еще не подтвержден.

`approved` - объект проверен и может использоваться как надежный источник.

`archived` - объект оставлен в Git для аудита, но исключен из derived catalogs,
индексов, RAG и external exports.

Smoke/test objects нужно архивировать после проверки pipeline.

## Проверка здоровья

Быстрая проверка:

```text
kb health
```

Расширенная серверная проверка:

```bash
python -m meeting_digest_bot knowledge-health --knowledge-dir /opt/company-knowledge
python -m meeting_digest_bot knowledge-quality-report --knowledge-dir /opt/company-knowledge
python -m meeting_digest_bot knowledge-rag-costs --knowledge-dir /opt/company-knowledge
```

Здоровое состояние:

- pending proposals = 0 или все ожидающие понятны;
- quality issues = 0;
- RAG chunks больше 0;
- Notion sync проходит без ошибок;
- Git working tree чистый после pipeline;
- backup timer активен.

## Как дальше уточнять сценарий использования

Когда ты будешь давать корректировку, лучше описывать ее как реальный рабочий
поток:

```text
Я хочу, чтобы после demo по задаче X бот делал Y,
затем спрашивал/не спрашивал подтверждение Z,
в Notion показывал это так,
а при запросе инструкции учитывал вот такие правила...
```

По такой корректировке можно будет обновить:

- правила отбора источников;
- структуру `task_case`;
- Notion views/properties;
- Telegram команды;
- RAG prompt modes;
- export profiles для NotebookLM и других AI.
